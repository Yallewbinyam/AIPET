**Helpers added on `AgentDevice`:**

```
classmethod active()         -- filter deleted_at IS NULL
classmethod with_deleted()   -- unfiltered

device.soft_delete(actor, reason=None)
    idempotent (no-op + no audit if already deleted -> returns False)
    audit: action='device.soft_deleted', node_meta carries
           {device_hostname, device_platform, reason, deleted_at}
    does NOT commit -- caller controls transaction

device.restore(actor, reason=None)
    idempotent (no-op + no audit if already active)
    audit: action='device.restored', node_meta carries
           {device_hostname, device_platform, reason,
            previously_deleted_at}

device.record_telemetry_after_delete(telemetry_at)
    audit-only (status='warning')
    Decision 2 of the brief: ACCEPT telemetry, DO NOT auto-restore,
    AUDIT the event for ops visibility (signal of either uninstall
    lag or key-reuse attack)
```

**Audit helper extended:** `iam/routes.py:log_action()` now takes
`details=None` kwarg that populates `audit_log.node_meta`.
Backward-compat (existing callers unchanged).

**Endpoints:**

* `DELETE /api/agent/devices/<id>` -- in-place conversion from hard
  to soft. JWT + per-tenant scoping. Body: `{reason?: string}`.
  Returns 200/`{deleted: true, deleted_at: ...}` on transition;
  200/`{deleted: true, already_deleted: true, deleted_at: ...}` on
  idempotent re-delete; 404 on unknown device or other-tenant
  device (no cross-tenant existence leak).

* `POST /api/agent/devices/<id>/restore` (NEW) -- mirror of DELETE.
  Returns 200 with restored serialisation; 200/`already_active=true`
  on idempotent restore; 404 on unknown.

* `GET /api/agent/devices?include_deleted=true` -- gated on
  `audit:read` permission OR `owner` role bypass (canonical
  pattern). Tenant scope unchanged: even an admin only sees
  deleted rows for their own user_id.

## Incidents found during recon

### Incident -- existing hard-delete cascade

The pre-existing `DELETE /api/agent/devices/<id>` endpoint did:

```python
AgentTelemetry.query.filter_by(agent_id=agent_id, user_id=uid).delete()
db.session.delete(device)
```

Hard delete of the device row AND mass-delete of every telemetry
row for that device. For a security platform that's a real issue:
it destroyed forensic data on every delete, and it ran without an
audit trail. The endpoint pre-dates this work (introduced before
the PLB-1 alembic baseline). It was reachable from the API but
NOT wired to the UI -- which is why no devices were ever actually
deleted in dashboard usage and the brief said "no DELETE endpoint
exists".

Resolution: in-place conversion to soft-delete preserves both the
device row (with `deleted_at`) and all telemetry rows (untouched).
Forensic continuity restored.

## Known issues / v1 limitations

### View Deleted access requires `audit:read` -- too broad for tenant self-service

The `?include_deleted=true` listing currently gates on the canonical
`audit:read` permission, which is intended for "view audit-grade
data" -- a fairly broad scope.

Concrete impact: a regular tenant user who soft-deletes a device by
accident has **no way through the UI to see or restore it in v1**.
They would need an admin (or `owner` role) to do it for them. This
is a real product gap, not just a UX inconvenience.

The right fix is a narrower scope -- something like
`device:read_deleted` granted by default to all tenant users, with
`audit:read` reserved for cross-tenant audit log viewing. Estimated
effort: 30 minutes (introduce the new permission, grant by default
in the role/permission seed, change the include_deleted check from
`audit:read` to `device:read_deleted OR audit:read`, update the
test that asserts the 403 path).

**Tracked as recommended follow-up below**, to be done before public
launch.

### datetime.utcnow() deprecation noise

The existing code uses `datetime.datetime.utcnow()` extensively;
new code in this work follows that convention to stay consistent.
Python 3.12 emits a DeprecationWarning. Switching to
`datetime.now(datetime.UTC)` is a project-wide cleanup task, not
in scope here.

## Recommendations (NOT closure items)

### Before public launch (priority order)

1. **Narrower scope for "View Deleted"** -- introduce
   `device:read_deleted` permission, grant by default, keep
   `audit:read` reserved for cross-tenant audit log access.
   ~30 min. Blocks tenant self-service for accidental deletes.

2. **Hard-delete-after-N-days policy** (optional) -- soft-deleted
   rows accumulate forever as written. Most security platforms
   eventually purge after a retention period (e.g. 7 years for
   compliance / 90 days for ops). Decide and add a Celery beat
   sweep when the team has bandwidth.

3. **Audit log indexes for action / resource lookups** -- as the
   table grows, queries like "show me all device.soft_deleted
   events for user X in the last 30 days" will start hitting full
   scans. Add a composite index `(action, timestamp DESC)` and
   possibly `(resource, timestamp DESC)` in a future migration.

### Pattern reuse

The `AgentDevice.active()` / `.with_deleted()` + lifecycle helpers
form a reusable pattern. Future models that need soft-delete should:

* Add `deleted_at TIMESTAMP NULL` (indexed)
* Add `active()` and `with_deleted()` classmethods
* Add `soft_delete(actor, reason=None)` and `restore(actor, reason=None)`
  -- both idempotent, both writing to `audit_log` via `log_action(...,
  details=...)`
* Audit every existing `Model.query` callsite during the conversion;
  no handwaves -- list them in the commit.

The `audit_log.node_meta` JSON column is the canonical place for
structured audit detail across the codebase, not just for
soft-delete.
